## Project structure

Iftarkar follows Next.js App Router conventions with a clear separation of concerns:

```
iftarkar/
├── app/                    # Next.js App Router
│   ├── layout.tsx          # Root layout with fonts & metadata
│   ├── page.tsx            # Home page with timer & duas
│   ├── calendars/[slug]/   # Dynamic calendar pages
│   │   └── page.tsx        # Calendar view with print support
│   ├── ical/[filename]/    # iCal export API routes
│   │   └── route.ts        # Generate .ics files
│   └── manifest.ts         # PWA manifest configuration
├── components/             # React components
│   ├── timer.tsx           # Main countdown timer
│   ├── settings.tsx        # Settings modal
│   ├── navbar.tsx          # Navigation bar
│   ├── modal.tsx           # Reusable modal components
│   ├── duas.tsx            # Dua cards container
│   └── ...
├── lib/                    # Utilities and business logic
│   ├── utils.ts            # Time calculations & helpers
│   └── settings.ts         # Settings class (localStorage)
├── data/                   # Static data
│   └── timings.json        # Prayer timings for all methods
├── public/                 # Static assets
│   ├── icons/              # PWA icons
│   └── logo_horizontal.svg # Brand logo
└── tailwind.config.ts      # Tailwind customization
```

## Architecture decisions

### Static site generation (SSG)

Iftarkar uses Next.js static export for maximum reliability:

<CodeGroup>
```mjs next.config.mjs
import withPWAInit from "@ducanh2912/next-pwa";

const withPWA = withPWAInit({
  dest: "public",
  disable: process.env.NODE_ENV === "development",
  register: true,
  skipWaiting: true,
});

const nextConfig = {
  output: "export",  // Static HTML export
};

export default withPWA(nextConfig);
```
</CodeGroup>

<Note>
**Why static export?** During Ramadan, external APIs can be unreliable. By pre-generating all pages and embedding timings data, Iftarkar guarantees 100% uptime even if the server goes down.
</Note>

### Progressive Web App (PWA)

The app is installable on mobile devices with offline support:

- Service worker caches all assets
- Works without internet after first load
- Native app-like experience
- Custom splash screens and icons

### Client-side state management

No external state management library needed:

<Tabs>
  <Tab title="Settings persistence">
    ```ts lib/settings.ts
    export class Settings {
      static prefix = "settingsV2-";
      
      static get method(): string {
        if (typeof window == "undefined") return Object.keys(timings)[0];
        return (
          localStorage.getItem(Settings.prefix + "timing") ??
          Object.keys(timings)[0]
        );
      }
      
      static set method(value) {
        localStorage.setItem(Settings.prefix + "timing", value);
      }
    }
    ```
  </Tab>

## Rendering strategy

### Hybrid approach

<Accordion title="Static pages">
- Home page (`/`)
- All calendar pages (`/calendars/raheemiya`, `/calendars/etk`, etc.)
- Pre-generated at build time using `generateStaticParams()`
</Accordion>

<Accordion title="Dynamic routes">
- iCal export API (`/ical/[filename]/route.ts`)
- Handles runtime generation of `.ics` calendar files
</Accordion>

<Accordion title="Client components">
All interactive components use `"use client"` directive:
- Timer with live countdown
- Settings modal
- Navigation menu
- Print buttons
</Accordion>

## Performance optimizations

### Hydration safety

The timer component prevents hydration mismatches:

```tsx components/timer.tsx
const [mounted, setMounted] = useState(false);

useEffect(() => {
  setMounted(true);
}, []);

if (!mounted) {
  return (
    <div className="w-full space-y-4 animate-pulse">
      {/* Skeleton UI */}
    </div>
  );
}
```

### Image optimization

Next.js Image component for optimized loading:

```tsx
import Image from "next/image";
import logo from "@/public/logo_horizontal.svg";

<Image
  className="h-8 md:h-10"
  src={logo}
  alt="Iftarkar Logo"
/>
```

### Code splitting

Automatic code splitting by route:
- Home page bundle
- Calendar page bundle
- Shared component chunks
